# API Integration Specifications: DocuMind AI

DocuMind AI interacts with several external APIs to orchestrate its autonomous documentation pipeline. This document outlines the key integrations.

## 1. GitHub / GitLab APIs
Used for monitoring repository events and automating Git operations.

### Webhook Integration
- **Endpoint**: `/api/v1/webhooks/github` (Internal DocuMind server route)
- **Purpose**: Listens for `push` and `pull_request` events.
- **Payload Processing**: Extracts commit hashes, modified file paths, and author metadata to trigger the `scanner.py` module.

### Pull Request Automation
- **API Target**: `https://api.github.com/repos/{owner}/{repo}/pulls`
- **Authentication**: GitHub App Installation Token or Personal Access Token (PAT).
- **Usage**: Once `generator.py` finishes updating the markdown files, the Git Manager opens a PR.
- **Request Payload Example**:
  ```json
  {
    "title": "docs: auto-update API and Architecture documentation",
    "body": "### Generated by DocuMind AI\n\n- Updated `API_SPECS.md` based on changes in `routes.py`\n- Synchronized `ARCHITECTURE.md` with new database schema.",
    "head": "documind/auto-update-1245",
    "base": "main"
  }
  ```

## 2. LLM Inference APIs
DocuMind relies on Large Language Models for semantic reasoning and natural language generation.

- **Primary Provider**: Groq (for high-speed inference) or Gemini/OpenAI (for complex reasoning).
- **Model Parameters**:
  - `temperature: 0.2` (Prioritizes factual, structured generation over creativity)
  - `max_tokens: 4000` (Allows generation of extensive markdown documents)

### System Prompt Context
The API requests heavily utilize system prompts structured to enforce developer-friendly output:
> "You are an autonomous technical documentation agent. Analyze the following code diff and architectural context. Update the provided markdown documentation. Preserve existing developer intent, do not hallucinate endpoints, and output raw markdown format."

## 3. Vector Database API
Used by the `memory.py` module to maintain codebase context.

- **Provider**: Azure Cognitive Search, Pinecone, or Qdrant.
- **Data Schema**:
  - `id`: File path or chunk identifier
  - `content`: Code snippet or docstring
  - `contentVector`: High-dimensional embedding (e.g., 384 or 1536 dimensions)
  - `metadata`: module name, last modified date
- **Operations**:
  - `Upsert`: Updating the vector DB when code changes are detected.
  - `Query`: Performing semantic similarity searches to provide contextual grounding for the LLM during generation.
